# MongoDB Migration - Complete Change Log

**Date:** April 14, 2026  
**Purpose:** Migrate from MySQL + Drizzle ORM to MongoDB  
**Status:** ✅ Complete - Local MongoDB Ready

---

## Overview

This document tracks all code, configuration, and dependency changes made to convert the ATS Resume Analyzer from a MySQL/Drizzle-based system to MongoDB.

**Key Changes:**
- Replaced Drizzle ORM with MongoDB driver
- Removed MySQL dependencies
- Converted SQL schema to MongoDB collections
- Updated environment configuration
- Maintained API compatibility

---

## 1. Dependency Changes

### Added
- `mongodb@6.6.0` - MongoDB Node.js driver

### Removed
- `mysql2` - MySQL client (no longer needed)
- `drizzle-orm` - ORM layer (replaced by MongoDB driver)

## 3. Core Database Layer

### File: `server/db.ts` (Complete Rewrite)
**Purpose:** Data access layer for all database operations

**Changes:**
1. **Connection Management**
   - Changed from Drizzle to MongoDB native driver
   - Lazy connection pattern preserved
   - Uses `MONGODB_URI` environment variable

2. **Collection Definitions**
   - `users` - User accounts and OAuth data
   - `resumes` - Uploaded resume files
   - `analyses` - ATS analysis results
   - `rewriteSuggestions` - LLM-generated suggestions

3. **Function Signatures (API-Compatible)**
   - `getDb()` - Returns MongoDB client
   - `upsertUser(user)` - Creates or updates user
   - `getUserByOpenId(openId)` - Queries user by OAuth ID
   - `createResume(...)` - Inserts resume document
   - `getResumesByUserId(userId)` - Queries resumes for user
   - `getResumeById(resumeId)` - Gets single resume
   - `createAnalysis(analysisData)` - Inserts analysis
   - `getAnalysisByResumeId(resumeId)` - Queries analysis
   - `getAnalysisById(analysisId)` - Gets single analysis
   - `createRewriteSuggestion(suggestionData)` - Inserts suggestion
   - `getRewriteSuggestionsByAnalysisId(analysisId)` - Queries suggestions

**Key Implementation Details:**
- Uses MongoDB transactions for data integrity
- ObjectId automatically generated by MongoDB for all documents
- Collections auto-created on first write
- Indexes created for frequently queried fields

## 7. Data Schema Changes

### Collections Structure

#### `users` (OAuth & Authentication)
```javascript
{
  _id: ObjectId,
  openId: String (unique),
  name: String | null,
  email: String | null,
  loginMethod: String | null,
  role: String ("user" | "admin"),
  createdAt: Date,
  updatedAt: Date,
  lastSignedIn: Date
}
```

#### `resumes` (Uploaded Files)
```javascript
{
  _id: ObjectId,
  userId: ObjectId (reference to users),
  fileName: String,
  fileKey: String | null,
  fileUrl: String | null,
  rawText: String (BSON max 16MB),
  createdAt: Date,
  updatedAt: Date
}
```

#### `analyses` (ATS Scoring Results)
```javascript
{
  _id: ObjectId,
  resumeId: ObjectId (reference to resumes),
  userId: ObjectId (reference to users),
  internScore: Number,
  jobScore: Number,
  matchedKeywordsIntern: String[],
  missingKeywordsIntern: String[],
  matchedKeywordsJob: String[],
  missingKeywordsJob: String[],
  checksCompleted: String[],
  passedChecks: String[],
  recommendedImprovements: String[],
  createdAt: Date,
  updatedAt: Date
}
```

#### `rewriteSuggestions` (LLM-Generated Improvements)
```javascript
{
  _id: ObjectId,
  analysisId: ObjectId (reference to analyses),
  suggest_bullet: String,
  original_bullet: String,
  improvement_type: String,
  createdAt: Date,
  updatedAt: Date
}
```

## 10. Compatibility Notes

### ✅ No Breaking Changes to APIs
- All tRPC procedures maintain same signatures
- All REST endpoints `/api/upload`, `/api/analyze` work unchanged
- All frontend hooks work unchanged

### ⚠️ Known Differences
1. **IDs**: Now MongoDB ObjectIds instead of auto-increment integers
   - Frontend handles them as strings transparently
   - API serialization handles conversion

2. **Transactions**: MongoDB provides ACID guarantees
   - No explicit transaction needed in simple operations
   - Multi-document transactions available if needed

3. **Indexes**: Not explicitly created
   - Should be added manually in production:
   ```javascript
   db.users.createIndex({ openId: 1 }, { unique: true })
   db.resumes.createIndex({ userId: 1 })
   db.analyses.createIndex({ userId: 1 })
   db.rewriteSuggestions.createIndex({ analysisId: 1 })
   ```

2. Create a cluster
3. Create database user with strong password
4. Whitelist IP address
5. Get connection string: `mongodb+srv://user:pass@cluster.mongodb.net/resume_optimizer_ats`

## 13. Rollback Plan (If Needed)

If reverting to MySQL is required:
1. Restore `drizzle/schema.ts` from git history
2. Restore `server/db.ts` to use Drizzle
3. Reinstall MySQL dependencies: `npm install mysql2 drizzle-orm drizzle-kit`
4. Update `MONGODB_URI` back to `DATABASE_URL` in `.env.local`
5. Run migrations: `pnpm drizzle-kit migrate`

## 14. Questions?

**Common Issues:**

**Q: "Cannot connect to MongoDB"**  
A: Ensure MongoDB is running (`mongod` for local) and URI is correct in `.env.local`

**Q: "TypeError: Invalid URL in OAuth"**  
A: Ensure `VITE_OAUTH_PORTAL_URL` and `VITE_APP_ID` are set in `.env.local`

**Q: "Module not found errors"**  
A: Run `npx pnpm install` to ensure dependencies are installed

**Q: "Old data is missing after migration"**  
A: MongoDB data is separate from MySQL. Both can coexist. Use old DATABASE_URL env var to access MySQL data if needed.
